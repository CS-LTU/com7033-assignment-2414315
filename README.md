## Security Implementation

### 1. Password Hashing (`pbkdf2:sha256`)
All passwords are hashed with Werkzeug's `generate_password_hash` using `pbkdf2:sha256` and a 16-byte random salt. Plaintext passwords are never stored or logged anywhere.

### 2. CSRF Protection
Every state-changing form includes a hidden CSRF token generated by **Flask-WTF**. The delete action uses a `POST` form (never a GET link) so it cannot be triggered by a malicious external request.

### 3. Input Validation & Sanitisation
- **WTForms validators** enforce field types, value ranges, and required fields on every form submission, server-side.
- **`Patient.validate()`** independently validates all fields before any MongoDB write — this acts as a second layer even if WTForms is bypassed.
- Strong password policy (uppercase + lowercase + digit + special character) enforced at registration.

### 4. Secure Session Handling
```python
SESSION_COOKIE_HTTPONLY = True    # JS cannot access session cookie
SESSION_COOKIE_SAMESITE = "Lax"  # Blocks cross-site request forgery
PERMANENT_SESSION_LIFETIME = 1800 # Sessions expire in 30 minutes
login_manager.session_protection = "strong"  # Regenerates on IP change
```

### 5. SQL Injection Prevention
All SQLite queries use **parameterised statements** (`?` placeholders). No string interpolation is used in any SQL query.

### 6. Open Redirect Prevention
The `next` redirect parameter after login is validated to be a relative path. Absolute URLs are rejected.

### 7. Generic Authentication Error Messages
Login failures always return `"Invalid username or password."` regardless of whether the username exists, preventing user enumeration attacks.

### 8. Error Logging
A **rotating file log** (`instance/logs/stroke_app.log`, max 1MB × 5 files) records all WARNING+ events with timestamps and module context. Sensitive data is never written to logs.

### 9. Security HTTP Headers (meta)
Base template includes `X-Content-Type-Options: nosniff` and `X-Frame-Options: SAMEORIGIN` meta tags.

## Importing the Dataset

Place `healthcare-dataset-stroke-data.csv` in the project root folder, then:

```bash
# Standard import (adds to existing records)
uv run python import_dataset.py

# Import with a custom file path
uv run python import_dataset.py --file path/to/healthcare-dataset-stroke-data.csv

# Clear all existing records first, then import fresh
uv run python import_dataset.py --clear
```

*(Note: If you used `./setup.sh`, the dataset was already imported automatically!)*

## Design Rationale

**Why two databases?**
SQLite is ideal for user authentication — it is relational, file-based, and fast for simple lookups. MongoDB is used for patient records because the Stroke Prediction Dataset is semi-structured and MongoDB's aggregation pipeline enables efficient statistics queries for the dashboard without complex SQL joins.

**Why the application factory pattern?**
`create_app()` allows the same codebase to run in development, production, and test modes with different configurations. It also makes the app fully testable without side effects.

**Why Blueprints?**
Blueprints enforce a separation of concerns — authentication, patient management, and dashboard are independent, testable modules. This follows the Single Responsibility Principle and makes the codebase scalable.

**Why WTForms over manual parsing?**
WTForms provides declarative, reusable validation that runs server-side regardless of whether client-side JavaScript is enabled. Combined with CSRF tokens it protects against injection and forgery simultaneously.

## Ethical Considerations

- **Data minimisation:** Only dataset-defined fields are stored. No extra personal data is collected.
- **Access control:** All patient data routes require authentication. Anonymous access is blocked.
- **Audit trail:** All logins, creations, edits, and deletions are logged with username and timestamp.
- **Healthcare data sensitivity:** In a real deployment, HTTPS (TLS) must be enforced and `SESSION_COOKIE_SECURE = True`. This would be set via a production `.env` file.
- **GDPR alignment:** Data is accessed only by authenticated staff, not shared with third parties, and access is restricted to the minimum necessary.
- **Password security:** Strong password requirements protect accounts even if a user chooses a simple base word.
